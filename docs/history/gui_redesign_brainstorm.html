02 · Design principles

Six commitments for the redesign

Every concept below is downstream of these. They're the rubric I'd hold the build to.

1

Show the activation, don't print it

Color-tinted token heatmaps, horizontal feature bars, before/after token diffs, diverging contrast bars. Magnitude should be a thing your eye catches in 200ms, with the exact float one hover away.

2

Make the loop the spine

Explore → Hypothesize → Steer → Measure → Save. The unit you carry through the app is a feature or a behavior, not a tab. Clicking a feature in Inspect should arm it in Steer; a good steer should flow into Bench; a passing bench should mint a recipe.

3

Meaning over magnitude

Lead with the human label, the top-activating tokens, the one-line "fires on…" summary. Keep raw IDs and floats present but secondary. A feature should introduce itself before it shows its math.

4

Always-present context

A persistent rail: model, layer, dtype, device, GPU memory, what's loaded, and the feature/recipe currently in focus. You should never wonder "what am I even looking at right now."

5

Honesty is a feature, so visualize it

The seven-method control story is the scientific heart of this tool. Render it as a scoreboard with a plain-English verdict — so a user sees why a recipe is "candidate," not "validated," and trusts the result more for it.

6

Delight earns the next click

Smooth strength sweeps, a satisfying steering dial, live diffs, a feeling of operating an instrument. Not decoration — momentum. The reward for exploring should be the joy of exploring more.

03 · Three design directions

Three lenses on the same workbench

Each is a complete, internally-consistent vision. They aren't mutually exclusive — the recommendation at the end fuses them — but seeing them apart sharpens the choice. Click to switch.

DIRECTION A
The Lab Bench
Workflow-first. One canvas, the loop as spine.
DIRECTION B
Feature Atlas
Exploration-first. Wander a space of features.
DIRECTION C
Steering Studio
Causal-first. The intervention is the hero.

The Lab Bench

Kill the tab strip. Replace it with a three-zone canvas: a left rail for navigation + live context, a center stage for the active experiment, and a right inspector that carries the feature or recipe you're working on across every step.

Navigation becomes the scientific loop itself — Explore · Steer · Measure · Library — so the tool teaches its own method just by existing. The thing you select stays selected as you move through stages.

Feature Atlas

Treat the SAE's tens of thousands of features as a place to wander, not a lookup table. The home screen is a gallery of feature cards — each with its id, human label (when known), a sparkline of its top-token activations, and a one-line "fires on…" summary.

Search and filter by label, layer, or activation profile. Click a card to drop into a feature detail with where-it-fires, a contrast view, and a "steer with this" button. The notebook stops being a tab and becomes margin notes on the card.

Steering Studio

Put the intervention dead center. A big strength dial, a live before/after token diff as you turn it, and an evidence panel — hook fired, hidden Δ, logits Δ — rendered as gauges instead of JSON keys.

The strength sweep becomes a filmstrip: the same prompt generated at strength −10 … 0 … +10, side by side, so the causal effect is something you scrub through. Below it, the benchmark "did it really work?" scoreboard with a one-glance verdict.

05 · The platform question

How beautiful can we actually get?

This is the one real architecture decision. The good news: SteeringService already returns plain dicts for every operation, so any of these reuses 100% of the engine.

ApproachVisual ceilingEffort & riskKeeps Modal / testsVerdict
A · Themed Gradio
Custom theme + CSS + gr.HTML for heatmaps/cards/diffs		 Medium — real upgrade, but capped by Gradio's layout model Low — incremental on app.py ✓ unchanged Fast win, low risk
B · FastAPI + custom SPA
Thin JSON API over the service, hand-built frontend		 High — full freedom; everything in this doc is buildable Medium — new API layer + frontend, new tests ~ Modal serve path changes (still ASGI) Highest ceiling
C · Phased hybrid
Prototype the experience static-first, then wire to a thin API; keep Gradio alive during transition		 High — same ceiling as B, de-risked Staged — value at every step ✓ Gradio stays until parity Recommended

Recommendation: fuse the directions, build it phased

The three directions aren't a vote — they're three rooms of one workbench. The Lab Bench is the shell (rail + stage + inspector). Feature Atlas is what "Explore" looks like. Steering Studio is what "Steer + Measure" looks like. One coherent app, three modes.

On platform: start with a static, mock-data HTML prototype of the Lab Bench shell so we lock the experience fast and cheaply — then wrap SteeringService in a thin FastAPI and wire it up surface by surface, keeping the existing Gradio app serveable until the new one reaches parity. Maximum ceiling, value at every step, nothing thrown away.
